# **ローカルLLMセンサーチャットシステム 設計書（Raspberry Pi スタンドアロン構成）**

## **1\. システム概要**

本システムは、ユーザーが「盆栽と会話しているように感じられる」体験を提供するための、ローカルLLM連携チャットシステムです。

最大の特徴は、外部のPCやクラウドサーバーに依存せず、**Raspberry Pi 5 (8GB) 単体ですべての処理を完結させるオールインワン構成**である点です。

デバイス内で「LLM推論」「センサーデータ収集」「記憶ロジック処理」「データベース」「Web UIホスティング」の全モジュールを稼働させます。温湿度、土壌水分、照度などの園芸向けセンサー値を取得し、盆栽の状態として会話コンテキストへ反映しながら、リアルタイムなストリーミング応答を実現します。

## **2\. システム構成・ハードウェア要件**

すべてのコンポーネントを同一デバイス上で稼働させるため、ハードウェアの安定性がシステム全体のパフォーマンスに直結します。

### **2.1 ハードウェア要件 (Raspberry Pi 5\)**

- **本体:** Raspberry Pi 5 (8GB RAM 必須)
  - LLMのモデル展開、インメモリ処理、フロントエンド/バックエンドの常駐でメモリを極限まで使用するため、8GBモデル以外は不可。
- **ストレージ:** MicroSDカード
  - データベースへの高頻度なログ書き込みや、LLMモデルのロード速度向上のため、SSDブートを推奨。

### **2.2 ソフトウェア・ネットワーク要件**

- **OS:** Raspberry Pi OS (64-bit)
- **稼働方式:** Docker Compose または systemd を用いて全モジュールをローカル環境で常時稼働。
- **ネットワーク:** すべてのコンポーネント間は localhost (127.0.0.1) 経由の REST API (HTTP/JSON) で通信。ユーザーは同一Wi-Fiネットワーク内のスマホやPCからラズパイのIPアドレスへアクセス。

## **3\. アーキテクチャ設計 (各モジュールの詳細)**

メモリ制限（8GB）の中で各プロセスを共存させるため、外部サーバー構成時に比べて軽量な技術スタックを選択します。

### **① LLM API サービス (LLM Node)**

- **機能:** 受け取ったプロンプトに基づきテキストを生成し、ストリーミングで返すAPIサーバー。
- **技術:** llama.cpp のサーバーモード。
  - **モデル:** Qwen 3B (GGUF形式、4bit量子化等を推奨)。
  - **制約:** メモリ使用量を約3〜4GBに抑えるため、コンテキストサイズ（n_ctx）を最適化して起動。

### **② センサーAPI サービス (Sensor Node)**

- **機能:** 定期的にセンサー値を読み取りメモリにキャッシュ。GETリクエストに対し即座にJSONで返す。
- **技術:** Python (FastAPI または軽量な Flask) \+ smbus2。
  - 軽量化のため、非常に小さなフットプリントで動作させる。

### **③ 記憶処理 & データベース (Memory Logic & DB Node)**

- **機能:** テキストのベクトル化（埋め込み）と、ベクトル検索を含むデータの永続化。
- **技術:** Python \+ 軽量埋め込みモデル \+ **SQLiteベースのDB**
  - **DB選定の変更点:** 重い ChromaDB や Milvus の代わりに、リソース消費が極めて少ない **SQLite \+ sqlite-vec**（ベクトル検索拡張）や、FAISS をインメモリで動かし定期的にファイルダンプする方式を採用。メモリ消費を数百MB単位に抑える。

### **④ バックエンドサーバー (Go Backend / BFF)**

- **機能:** システム全体のオーケストレーション。Next.js(UI)からのリクエストを受け、ローカル内のセンサー、記憶DBに並行して問い合わせを行い、プロンプトを構築してLLM APIへ投げる。
- **技術:** Go (Golang)
  - 非常に軽量で高速なため、ラズパイ上でもCPU/メモリ負荷をほとんどかけずに並行処理（goroutine）が可能。

### **⑤ Web UI サーバー (Frontend)**

- **機能:** ユーザーがブラウザからアクセスするチャット画面の提供。
- **技術:** Next.js (React) または軽量な代替（Preact / Vanilla JS）。
  - **稼働方式の変更点:** Next.jsのNode.jsサーバーを常駐させるとメモリを食うため、**静的エクスポート (Static HTML Export)** 機能を使い、ビルド済みのHTML/JS/CSSファイルを ④のGoバックエンドから静的に配信 (Serve) する構成を強く推奨。これによりWeb UI用のサーバープロセスをゼロにできる。

## **4\. データフロー（会話発生時の通信フロー）**

通信はすべてラズパイの内部(localhost)で行われます。

1. **\[User \-\> Go Backend\]** ユーザーがスマホ等のブラウザからラズパイのIPにアクセスし、チャットを入力。Goバックエンドがリクエストを受信。
2. **\[Go Backendによる並行データ収集 (内部通信)\]** goroutineで以下を同時に実行。
   - **\-\> Sensor API:** localhost:xxxx へ環境データをリクエスト。
   - **\-\> Memory & DB:** localhost:yyyy へ入力を送信し、関連する記憶ベクトルを検索・取得。
3. **\[Go Backend \-\> LLM API (内部通信)\]** 収集データを統合し、プロンプトを構築して localhost:zzzz の llama.cpp サーバーへ送信。
4. **\[LLM API \-\> Go Backend \-\> User\]** Qwen 4Bが推論実行。Server-Sent Events (SSE) でGoを経由し、ユーザーのブラウザへストリーミング返答。
5. **\[Go Backend \-\> DB (非同期)\]** 会話完了後、会話履歴の保存を Memory & DB ノードへ非同期でリクエスト。

## **5\. 懸念点と対処法（優先度別）**

単体構成ならではのハードウェア・リソースの限界に対する対策が最優先事項となります。

### **5.1 優先度: 高 (システム稼働の致命傷を防ぐ)**

1. **メモリ(RAM)の枯渇 (Out Of Memory: OOM)**
   - **懸念:** LLM、DB、各種APIが8GBのメモリを奪い合い、プロセスが強制終了（OOM Killer）される。
   - **対処法:**
     - LLMのモデルサイズ（量子化ビット数）とコンテキストサイズを厳密に制限（LLMに最大4GB、OSとその他に4GB等）。
     - Next.jsをNode.jsサーバーで動かすのをやめ、静的ファイルをGoから配信してメモリを節約。
     - Swapメモリ（ZRAMなど）を適切に設定し、クラッシュを緩衝する。
2. **ストレージ(MicroSD)の短寿命化とI/Oボトルネック**
   - **懸念:** DBへの会話ログ書き込みや、Swap発生時の書き込み過多により、MicroSDカードが早期に寿命を迎え破損する。
   - **対処法:**
     - **NVMe SSD \+ PCIe HAT** の導入を強く推奨（速度・耐久性が劇的に向上）。
     - MicroSD運用を強行する場合は、DBの書き込みをオンメモリ（Redis等）でバッファリングし、数時間に1回ファイルへ一括同期（バルクインサート）するよう設計変更。
3. **熱暴走と電源不足**
   - **懸念:** LLM推論時にCPUが100%に張り付き、熱でクロックダウン（応答遅延）が起きる。または電源容量不足で再起動する。
   - **対処法:**
     - 純正27W電源と、大型ヒートシンク付きファン（アクティブクーラー）を物理的に必ず装備する。

### **5.2 優先度: 中 (UXと運用性の向上)**

1. **推論速度（トークン生成速度）の低下**
   - **懸念:** ラズパイ単体のため、外部GPUサーバーのような高速な返答は期待できず、ユーザーが「遅い」と感じる。
   - **対処法:**
     - UI側で「盆栽が考えている」ようなアニメーションやシステムメッセージを表示し、体感待ち時間を緩和する。
     - ストリーミング表示（SSE）を確実に実装し、最初の1文字目（TTFT）をいかに早く出すかに注力する。
2. **複数プロセス管理の煩雑さ**
   - **懸念:** 4つのサービスが動くため、どれか一つが落ちた時の復旧が面倒。
   - **対処法:**
     - docker-compose.yml で全サービスを一元管理し、restart: unless-stopped を設定。
     - リソース制限（deploy.resources）をDocker側でかけ、特定のコンテナがメモリを食いつぶさないようにする。

### **5.3 優先度: 低 (将来の拡張性)**

1. **将来的なモデルのスケールアップ困難**
   - **懸念:** 今後より賢いモデル（8Bクラス等）が出た際、ラズパイの8GBメモリでは載せ替えられない。
   - **対処法:**
     - まずはこの構成（4Bモデル等）で「盆栽との会話の質」がユースケースを満たすか検証(PoC)する。
     - 万が一LLMの知能が不足する場合は、LLM API部分のみを外部API（OpenAI等）または別PCへオフロードできるよう、BFF(Go)側のインターフェースを疎結合に保っておく。
